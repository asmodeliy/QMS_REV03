📋 CITS 개요

CITS (Customer Issue Tracking System)는 고객으로부터 들어오는 문제(이슈)를 한곳에서 추적하고 관리하는 통합 시스템입니다.

주요 특징

  • 📥 이슈 수집: 고객 이슈를 빠르게 등록하고 추적합니다.
  • 🔄 상태 관리: OPEN → PENDING → CLOSE 3가지 상태로 진행도를 관리합니다.
  • 👥 담당자 지정: 각 이슈를 팀원에게 할당하여 책임을 명확히 합니다.
  • 📎 첨부파일: 고객 보내온 로그, 메모리 덤프, 스크린샷 등을 첨부합니다.
  • 💬 커뮤니케이션: 내부 팀과 고객 간 커뮤니케이션을 기록합니다.

🚀 빠른 시작

1단계: 새 이슈 등록

  • 우측 상단의 "+ New Issue" 버튼을 클릭합니다.
  • 고객 정보와 문제를 입력합니다:
    • 고객명: 문제를 보고한 고객
    • 제목: 문제의 간단한 설명
    • 설명: 상세한 문제 상황, 재현 방법, 영향도
    • 제품: 문제가 발생한 제품/칩
    • 노드: 해당 노드
  • 고객이 보낸 첨부파일 (로그, 덤프 등)을 첨부합니다.
  • "Create" 버튼을 클릭합니다.

2단계: 이슈 분석 및 할당

  • 등록된 이슈에서:
    • 내용을 검토하고 담당팀/담당자를 지정합니다.
    • 필요시 고객에서 추가 정보를 요청합니다 (댓글 사용).

3단계: 이슈 진행

  • 상태를 "PENDING"으로 변경합니다.
  • 분석/수정 작업을 진행하며 주기적으로 댓글을 업데이트합니다.

4단계: 이슈 종료

  • 문제가 완전히 해결되면 상태를 "CLOSE"로 변경합니다.
  • 해결 방법과 결과를 최종 주석에 기록합니다.

🔄 이슈 워크플로우

3단계 상태 관리

OPEN → PENDING → CLOSE

상태 상세 설명

🔴 OPEN (열림) - 새로운 이슈

  • 방금 등록된 이슈 상태입니다.
  • 팀 리더나 담당자가 내용을 검토합니다.
  • 필요시 고객에게 추가 정보를 요청합니다.
  • 담당팀/담당자를 지정해야 합니다.

🟠 PENDING (대기중) - 진행 중

  • 분석 또는 수정 작업이 진행 중인 상태입니다.
  • 담당자가 주기적으로 진행 상황을 업데이트합니다.
  • 고객에게 진행 현황을 공유합니다.
  • 필요시 고객과 협력하여 문제를 해결합니다.

🟢 CLOSE (종료) - 해결됨

  • 이슈가 완전히 해결되고 고객이 검증한 상태입니다.
  • 해결 방법, 수정사항, 검증 결과를 상세히 기록합니다.
  • 고객에게 최종 해결 방법을 안내합니다.

상태 전환 규칙

  • OPEN → PENDING: 분석 시작 시
  • PENDING → CLOSE: 고객 검증 완료 후
  • 되돌리간: CLOSE에서 PENDING으로 되돌리칠 수 있습니다 (재현 시)

📝 필드 설명

고객 정보

  • 고객명: 문제를 보고한 고객 또는 조직
  • 연락처: 고객 담당자 연락처
  • 고객 이슈 ID: 고객이 자신의 시스템에서 사용하는 ID (선택사항)

기술 정보

  • 제품/칩: 문제가 발생한 제품 또는 칩 이름
  • 노드: IP/IC, Node 정보
  • 버전: 소프트웨어/펌웨어 버전
  • 현상: 발생한 문제의 구체적인 현상

관리 정보

  • 상태: OPEN / PENDING / CLOSE
  • 담당자: 이슈를 담당할 팀원
  • 심각도: Critical / High / Medium / Low (선택 사항)
  • 태그: 검색과 분류를 위한 태그

첨부파일

  • 고객이 보낸 로그, 메모리 덤프, 스크린샷 등을 첨부합니다.
  • 파일명에 특수문자(# ? %)는 사용하지 않습니다.
  • 대용량 파일은 zip으로 압축합니다.

✅ 베스트 프랙티스

이슈 등록시 팁

  • ✍️ 명확한 제목: "Bug" 대신 "Chip XX not responding after power reset"처럼 구체적으로
  • 📐 재현 방법 상세히: 다른 팀원도 쉽게 문제를 재현할 수 있도록
  • 📎 첨부파일 준비: 로그, 덤프, 스크린샷을 함께 제공
  • 🎯 심각도 정확히: 제품 출시, 고객 만족도에 미치는 영향을 고려

이슈 관리시 팁

  • ⏰ 빠른 응답: Critical 이슈는 4시간 내 응답
  • 🔄 정기적 업데이트: 최소 주 1회 진행 상황을 고객에게 알립니다
  • 📝 상세한 기록: 시도한 모든 방법, 결과를 기록합니다
  • ✅ 고객 검증: 고객이 해결을 확인한 후 CLOSE합니다

심각도 가이드

  • 🔴 Critical: 제품이 작동하지 않음, 데이터 손상
  • 🟠 High: 주요 기능이 작동하지 않음
  • 🟡 Medium: 특정 기능에 결함, 임시방안 있음
  • 🟢 Low: 부가 기능, 사소한 버그

❓ 자주 묻는 질문

Q: 이슈를 고객과 공유할 수 있나요?

A: 네, 이슈를 생성한 후 고객에게 이슈 ID와 링크를 전달하면 고객도 진행 상황을 확인할 수 있습니다. 관리자에게 고객 접근 권한을 요청하세요.

Q: 이슈를 재개(Reopen)할 수 있나요?

A: 네, CLOSE 상태의 이슈에서 "Reopen" 버튼으로 다시 PENDING 상태로 변경할 수 있습니다. 고객이 문제를 재현한 경우 사용합니다.

Q: 고객으로부터 온 로그가 너무 큽니다. 어떻게 하나요?

A: 고객에게 필요한 부분만 추출해달라고 요청하거나, zip으로 압축하여 여러 파일로 나누어 업로드하세요.

Q: 여러 고객 문제가 같은 원인일 수 있나요?

A: 네, 가능합니다. "Related Issues" 기능으로 연관된 이슈들을 링크할 수 있습니다. 관리자에게 연관 관계를 요청하세요.

Q: 해결 방법을 다른 팀과 공유하려면?

A: 이슈의 최종 댓글에 해결 방법을 상세히 기록하고, 필요한 팀원을 @태그로 언급하면 알림을 받게 됩니다.
